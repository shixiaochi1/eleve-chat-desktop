# Eleve 桌面前端断线诊断与修复方案

> 诊断时间：2026-07-20
> 诊断人：小虾米
> 范围：`eleve-chat-desktop` 前端 ↔ `eleve-gateway` 后端全链路

---

## 一、架构概览

```
Tauri v2 壳 (lib.rs)
  └─ 启动 eleved 子进程
  └─ 轮询 gateway_state.json 发现端口
  └─ 4 个 Tauri Command: get_gateway_port / get_auto_start / set_auto_start / resolve_media

前端三通道桥接 (bridge.ts)
  ├─ 会话类命令 → WS JSON-RPC (ws-client.ts, 63 个映射)
  ├─ 配置类命令 → HTTP REST (CONFIG_HTTP_MAP, 7 个映射)
  └─ Kanban 命令 → HTTP REST (KANBAN_HTTP_MAP, ~40 个映射)

后端 (eleve-gateway)
  ├─ WS: ws/mod.rs 注册 180+ 方法
  └─ HTTP: api_server.rs 注册 80+ 路由
```

## 三、修复方案

### 修复 1：渠道面板（P0）

**方案 A（推荐）：后端新增 channels API**

在 `eleve-gateway/src/api_server.rs` 新增：

```rust
// GET /api/channels — 返回已连接的平台渠道列表
.route("/api/channels", get(api_channels_handler))
```

Handler 从 `AppState` 读取已注册的 platform 连接状态，返回：

```json
{
  "channels": [
    {
      "id": "wechat-default",
      "name": "微信",
      "platform": "wechat",
      "status": "connected",
      "last_activity": "2026-07-20T07:00:00Z",
      "config": {}
    }
  ]
}
```

前端 `useChannels.ts` 已有完整的 fallback 链和 UI 渲染逻辑，只需后端提供数据即可。

**方案 B（临时）：前端改用 gateway_status 的 platforms 字段**

修改 `useChannels.ts`，跳过不存在的 `/v1/channels` 和 `/api/channels`，直接从 `call('gateway_status')` 的 `platforms` 字段提取渠道信息。

- 改动文件：`src/hooks/useChannels.ts`
- 改动量：~20 行

### 修复 2：文件浏览器（P0）

**方案：添加 tauri-plugin-path 依赖**

1. `src-tauri/Cargo.toml` 添加：

```toml
[dependencies]
tauri-plugin-path = "2"
```

2. `src-tauri/src/lib.rs` 注册插件：

```rust
tauri::Builder::default()
    .plugin(tauri_plugin_dialog::init())
    .plugin(tauri_plugin_single_instance::init(|app, _args, _cwd| { ... }))
    .plugin(tauri_plugin_path::init())   // ← 新增
    ...
```

3. `src-tauri/tauri.conf.json` 确认 plugins 配置（插件自动注册，通常无需额外配置）

4. 前端 `FileBrowserPanel.tsx` 的 `invoke('plugin:path|resolve_home_dir')` 调用无需修改

- 改动文件：`src-tauri/Cargo.toml`、`src-tauri/src/lib.rs`
- 改动量：~3 行
- ⚠️ 需要在 PowerShell/CMD 中重新编译：`cargo build -p eleve-chat-desktop`

### 修复 3：终端面板升级为交互式 PTY（P1）

**方案：对接后端已有的 terminal WS 方法**

后端已注册：
- `terminal.resize` — 调整终端尺寸
- `terminal.read.respond` — 读取终端缓冲区
- `shell.exec` — 执行 shell 命令（流式输出）
- `cli.exec` — 执行 CLI 命令
- `process.list` / `process.kill` / `process.stop` — 进程管理

前端需要：

1. 新建 `src/hooks/usePtySession.ts`：
   - 通过 WS `shell.exec` 创建 PTY 会话
   - 监听 WS 事件 `terminal.output` 接收输出流
   - 发送 `terminal.resize` 同步尺寸
   - 发送 `terminal.input` 传递用户键入

2. 改造 `TerminalPanel.tsx`：
   - xterm.js 的 `onData` → WS `terminal.input`
   - WS `terminal.output` → xterm.js 的 `write()`
   - 保留现有 `execute_command` 作为快捷命令模式

- 改动文件：新建 `usePtySession.ts`，改造 `TerminalPanel.tsx`
- 改动量：~200 行
- 前提：确认后端 `shell.exec` 支持 PTY 流式输出（需检查 `terminal_gateway.rs`）

## 五、诊断方法说明

本次诊断通过以下方法进行静态代码分析：

1. **WS 方法对比**：提取前端 `bridge.ts` 的 63 个 WS 映射 vs 后端 `ws/mod.rs` 的 180+ 注册方法，`comm -23` 确认 0 缺失
2. **HTTP 路由对比**：提取后端 `api_server.rs` 全部 `.route()` 注册，逐一比对前端 `CONFIG_HTTP_MAP` 和 `KANBAN_HTTP_MAP`
3. **组件 API 调用扫描**：grep 全部 70+ 组件的 `call()`、`fetch()`、`invoke()` 调用，确认数据源
4. **依赖检查**：核实 `Cargo.toml`、`tauri.conf.json` 的插件声明
5. **后端 handler 实现验证**：追踪 `update_config_raw`、`handle_command_dispatch` 等关键函数的参数处理逻辑

> ⚠️ 本次为静态分析，未实际运行前端。🟡 项需运行时验证确认。
